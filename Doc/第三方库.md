## libtiff

**libtiff 是一个开源的库**。它根据一个宽松的许可证发布，具体是 **libtiff 许可证**，该许可证与 BSD 许可证兼容。这意味着你可以在开源和闭源项目中自由使用、修改和分发它。

### 源码下载

你可以从以下几个官方或可靠的来源下载 libtiff 的源代码：

1.  **官方 GitLab 仓库（推荐）**
    这是 libtiff 项目目前最主要的开发托管地。你可以从这里获取最新的源代码。
    
    *   **主仓库地址**: [https://gitlab.com/libtiff/libtiff](https://gitlab.com/libtiff/libtiff)
    *   **下载方式**： 使用 `git` 克隆是最佳方式。
        ```bash
        git clone https://gitlab.com/libtiff/libtiff.git
        ```
    
2.  **官方发布版本（稳定版）**
    如果你希望使用经过测试的稳定版本，而不是最新的开发版，可以下载官方发布的 tar 包。
    
    *   **下载页面**: [http://download.osgeo.org/libtiff/](http://download.osgeo.org/libtiff/)
    *   在这个页面中，找到最新的版本（例如 `tiff-4.6.0.tar.gz`），点击下载。

**对于大多数用户，建议下载官方发布版本的 tar 包，因为它更稳定。**

---

### 编译指南

libtiff 使用标准的 Unix 构建系统 **Autotools**（`configure` 脚本）和 **Makefile**。在 Windows 上，你可以使用 Visual Studio 的解决方案文件或 MSYS2/MinGW 环境。

```
cmake -S . -B ../libtiff.build -G "Visual Studio 16 2019" -A x64 -DBUILD_SHARED_LIBS=ON -Dtiff-tools=OFF -DCMAKE_INSTALL_PREFIX=../libtiff.install 
cmake --build ../libtiff.build --config Debug --target install
```

### 重要的 CMake 配置选项

在 `libtiff/CMakeLists.txt` 中定义了这些开关（名字是固定的）：

| 功能                   | 选项                | 备注         |
| ---------------------- | ------------------- | ------------ |
| 构建动态库而不是静态库 | `BUILD_SHARED_LIBS` |              |
| 测试                   | `tiff-tests`        |              |
| 工具                   | `tiff-tools`        | 依赖 tiffgt  |
| JPEG 压缩              | `jpeg`              | 依赖 libjpeg |
| LZMA 压缩              | `lzma`              | 依赖 liblzma |
| WebP 压缩              | `webp`              | 依赖 libwebp |
| Zlib 压缩              | `zlib`              |              |
| ZSTD 压缩              | `zstd`              |              |

它们都是 **ON/OFF 类型 option()**，推荐在 cmake 命令行关闭。

例如你要关闭 `libjpeg-9__.dll、liblzma-5__.dll、libwebp-7__.dll` 这三个 dll 依赖：

```
cmake -S . -B build ^
  -Djpeg=OFF ^
  -Dlzma=OFF ^
  -Dwebp=OFF
```

如果你想更彻底（最小依赖版 libtiff）：

```
cmake -S . -B build ^
  -DBUILD_SHARED_LIBS=ON ^
  -Dtiff-tools=OFF ^
  -Dtiff-tests=OFF ^
  -Djpeg=OFF ^
  -Dlzma=OFF ^
  -Dwebp=OFF ^
  -Dzlib=OFF ^
  -Dzstd=OFF
```

然后：

```
cmake --build build
```

## simpleini

### 配置

在 `CMakeLists.txt` 中，你会看到这一行：

```CMake
add_library(${PROJECT_NAME} INTERFACE)
```

`INTERFACE` 库是 CMake 用来描述 "header-only" 库的“标准姿势”。它告诉 CMake：“我是一个库，我没有要编译的源文件，但任何链接我的目标都需要包含我的头文件。”

```cmake
# 1. 进入你下载的 SimpleIni 源码目录
cd E:\DemoProject\3rd-source

# 2. 运行 CMake 配置命令
#    -S simpleini          : 源码在当前目录
#    -B simpleini.build    : 指定构建目录为 "simpleini.build" (这个名字随意，但不能是 ".") 
#    -G "Visual Studio 16 2019" : 告诉 CMake 为 VS2019 生成项目
#    -DBUILD_TESTING=OFF : 告诉 CMake 构建测试，否则会去下载 googletest 导致配置出错
#    -DCMAKE_INSTALL_PREFIX=simpleini.install :  **这是关键！** 指定安装路径

cmake -S simpleini -B simpleini.build -G "Visual Studio 16 2019" -DBUILD_TESTING=OFF -DCMAKE_INSTALL_PREFIX=simpleini.install
```

### 安装

```CMake
cmake --install simpleini.build --config Release
```

## GoogleTest

### 执行 CMake 配置

```bash
cmake -S googletest -B googletest.build -G "Visual Studio 16 2019" -A x64  -DCMAKE_INSTALL_PREFIX=googletest.install -Dgtest_force_shared_crt=ON -DCMAKE_DEBUG_POSTFIX=d
```

**关键参数说明：**

- `-Dgtest_force_shared_crt=ON`：强制 gtest 使用动态链接运行时库 (`/MD` 或 `/MDd`)。这非常重要，因为 VS 默认项目通常使用 `/MD`，如果这里不开启，链接时会报 `LNK2038: 检测到“RuntimeLibrary”的不匹配项` 错误。
- `-DCMAKE_DEBUG_POSTFIX=d`：为 Debug 版本的库文件名自动添加一个后缀（通常是 "d"），否则 Debug 和 Release 需要 install 到不同的目录。

### 编译与安装

```bash
# 编译并安装 Debug 版本
cmake --build googletest.build --config Debug --target INSTALL
# 编译并安装 Release 版本
cmake --build googletest.build --config Release --target INSTALL
```

执行完成后，`googletest.install` 目录下会出现 `include` 和 `lib` 文件夹。

### 在测试程序中调用 SDK (CMake 方式)

这是最推荐的方法，因为它能自动处理复杂的依赖关系。

在你的测试项目根目录下编写 `CMakeLists.txt`：

```CMake
cmake_minimum_required(VERSION 3.14)
project(MyTestProject)

# 1. 设置 gtest 的安装路径
set(strGTestInstallRoot "${CMAKE_CURRENT_SOURCE_DIR}/../3rd-source/googletest.install")

# 2. 查找 GoogleTest 及其产生的映射文件
if(EXISTS "${strGTestInstallRoot}/lib/cmake/GTest/GTestConfig1.cmake")
    list(APPEND CMAKE_PREFIX_PATH "${strGTestInstallRoot}") 

    find_package(GTest CONFIG REQUIRED)
    message(STATUS "Using installed GoogleTest from unified directory: ${strGTestInstallRoot}")
else()
    message(STATUS "Local GoogleTest installation not found. Falling back to FetchContent.")
    include(FetchContent)
    FetchContent_Declare(
        googletest
        URL https://github.com/google/googletest/archive/refs/tags/v1.14.0.zip
        DOWNLOAD_EXTRACT_TIMESTAMP TRUE
    )
    set(gtest_force_shared_crt ON CACHE BOOL "" FORCE)
    FetchContent_MakeAvailable(googletest)
    message(STATUS "Using fallback GoogleTest targets from FetchContent.")
endif()

# 3. 添加你的测试源码
add_executable(CMyTestMain main.cpp)

# 4. 链接 gtest 库
# GTest::gtest_main 会自动包含 main 函数，省去你自己写 main 的麻烦
target_link_libraries(CMyTestMain PRIVATE GTest::gtest GTest::gtest_main)
```

------

### 在测试程序中手动配置 (Visual Studio IDE 方式)

如果你不使用 CMake 管理测试程序，而是直接在 VS 2019 属性页配置：

1. **头文件**：`项目属性 -> C/C++ -> 常规 -> 附加包含目录`，添加 `C:\gtest_sdk\include`。
2. **库目录**：`项目属性 -> 链接器 -> 常规 -> 附加库目录`，添加 `C:\gtest_sdk\lib`。
3. **库文件**：`项目属性 -> 链接器 -> 输入 -> 附加依赖项`：
   - **Debug 模式**：添加 `gtestd.lib` 和 `gtest_maind.lib`。
   - **Release 模式**：添加 `gtest.lib` 和 `gtest_main.lib`。
4. **运行时库**：确保 `项目属性 -> C/C++ -> 代码生成 -> 运行库` 设置为 **多线程 DLL (/MD)**（需与编译 SDK 时的设置一致）。

------

### 编写一个简单的测试示例

在 `main.cpp` 中尝试运行以下代码：

```C++
#include <gtest/gtest.h>

// 待测逻辑
int Add(int nLeft, int nRight) {
    return nLeft + nRight;
}

// 定义测试用例
TEST(CMathTest, AddFunction) {
    EXPECT_EQ(Add(1, 2), 3);
    EXPECT_EQ(Add(-1, 1), 0);
}

// 如果链接了 gtest_main，则不需要写 int main()
// 否则需要手动编写如下代码：
/*
int main(int argc, char **argv) {
    ::testing::InitGoogleTest(&argc, argv);
    return RUN_ALL_TESTS();
}
*/
```

------

### 注意事项

- **Debug/Release 匹配**：请务必确保你的测试程序版本（Debug/Release）与链接的 gtest 库版本一致。
- **C++ 标准**：GoogleTest 现在通常要求 C++14 或更高版本，请在 VS 项目属性中将 `C++ 语言标准` 设置为 `ISO C++14` 或 `ISO C++17`。

## Sentry

### clone 仓库

```cmake
cd E:\project\3rd-source\
git submodule add https://github.com/getsentry/sentry-native.git

# 初始化并更新所有子模块（这一步会真正去下载 crashpad 等源码）
cd E:\project\3rd-source\sentry-native
git submodule update --init --recursive
```

### 生成解决方案

运行 CMake 生成 Visual Studio 的解决方案（`.sln`）文件。在执行此命令时，有几个关键的配置项需要注意：

```Bash
cmake -S sentry-native -B sentry-native.build -G "Visual Studio 16 2019" -A x64 ^
  -DSENTRY_BACKEND=crashpad
```

**关键参数说明：**

- `-G "Visual Studio 17 2022" -A x64`：指定生成 VS2022 的 64 位工程（如果是 VS2019，请改为 `"Visual Studio 16 2019"`）。**请确保此处的架构（x64/x86）与你的 Qt 项目保持绝对一致。**
- `-DCMAKE_INSTALL_PREFIX="sentry-native.install"`：指定编译完成后，头文件和库文件的导出目录。
- `-DSENTRY_BACKEND=crashpad`：在 Windows 上，强烈推荐使用 `crashpad` 作为后端，它是谷歌开发的崩溃捕获工具，支持进程外（Out-of-process）捕获，比 `breakpad` 更稳定，且能捕获更严重的崩溃。
- **关于运行时库（MSVC Runtime）的重要提示**： 默认情况下，`sentry-native` 会采用动态链接运行时库（`/MD` 或 `/MDd`）。**这通常与 Qt 官方安装包的默认配置（MD/MDd）是一致的。** 如果你的 Qt 项目修改为了静态链接（`/MT` 或 `/MTd`），你需要向 CMake 传递以下参数来强制匹配： `-DCRASHPAD_MSVC_RUNTIME_LIBRARY=MultiThreaded` （对应 MT）

**注意事项：**

Sentry 内存在子模块，所以`-DCMAKE_DEBUG_POSTFIX=d`只对 Sentry 生效，子模块在 debug 下的输出仍然没有 `d` 后缀，我可以在执行 `install` 时，将 Debug 和 Release 输出到不同的安装路径中来解决。

### 执行编译与安装

配置完成后，直接使用 CMake 命令调用 Visual Studio 编译器进行构建。

- **编译 Debug 版本**（用于你的 Qt Debug 模式）：

  ```Bash
  cmake --build sentry-native.build -j --config Debug
  
  cmake --install sentry-native.build --config Debug --prefix "E:/project/3rd-source/sentry-native.debug.install"
  ```

- **编译 Release 版本**（用于你的 Qt Release 模式）：

  ```Bash
  cmake --build sentry-native.build -j --config Release
  
  cmake --install sentry-native.build --config Release --prefix "E:/project/3rd-source/sentry-native.release.install"
  ```

### 编译产物检查

编译完成后，在你指定的 `sentry-native.install` 目录下会生成以下结构：

- **`include/`**：包含 `sentry.h` 头文件（供 Qt 项目 `#include` 使用）。
- **`bin/`**：包含 `sentry.dll`、`crashpad_handler.exe`（重要：崩溃捕获服务程序）。
- **`lib/`**：包含 `sentry.lib` 导入库。

### 在 Qt 项目中接入 Sentry (CMake)

如果你的 Qt 项目也是基于 CMake 构建的，可以在你的 Qt 项目 `CMakeLists.txt` 中这样引入 Sentry：

CMake

```cmake
# 1. 设置 Sentry 路径（指向刚才 install 的目录）
set(Sentry_DIR "D:/path/to/sentry-native/install")

# 2. 包含头文件路径
target_include_directories(${PROJECT_NAME} PRIVATE "${Sentry_DIR}/include")

# 3. 链接库文件
target_link_libraries(${PROJECT_NAME} PRIVATE "${Sentry_DIR}/lib/sentry.lib")

# 4. 确保程序运行时能找到 DLL 和 crashpad_handler.exe
# 在构建完成后，自动将 bin 目录下的文件复制到 Qt 的可执行文件输出目录
add_custom_command(TARGET ${PROJECT_NAME} POST_BUILD
    COMMAND ${CMAKE_COMMAND} -E copy_directory
    "${Sentry_DIR}/bin"
    "$<TARGET_FILE_DIR:${PROJECT_NAME}>"
)
```
